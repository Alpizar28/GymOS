## Project Overview

GymOS is a single-tenant personal training system for one athlete. It combines:
- A **FastAPI + SQLAlchemy** async backend (Python 3.11+)
- A **Next.js 16 + React 19 + Tailwind** frontend
- A **deterministic RIR-based progression engine**
- **OpenAI GPT-4o** for plan generation
- **SQLite** (aiosqlite) as the database

The true project root is the `GymOS/` directory (one level of nesting from the repo root).
All commands below are run from that directory unless noted.

ruff format src/

## Python Code Style

### Async-First
Every service function, DB query, and route handler must be `async def`. Use
`AsyncSession` from SQLAlchemy with `async with async_session() as session:`.

### Type Hints
All function signatures must have full type hints. Use the Python 3.10+ union
syntax: `int | None`, `str | None` (not `Optional[int]`).

### Imports
Ruff enforces `isort` (rule `I`). Order: stdlib → third-party → `src.*`.
Inline imports inside function bodies are acceptable for optional or circular
dependencies.

### Naming
- Functions and variables: `snake_case`
- Classes: `PascalCase`
- Constants: `UPPER_SNAKE_CASE`
- Database columns: `snake_case`
- REST route paths: `kebab-case` (e.g. `/generate-today`, `/week-template`)

### Error Handling
- Routes raise `HTTPException` with explicit status codes and detail strings.
- Service functions return `None` on soft failures (do not raise). The caller
  checks for `None` and raises `HTTPException` if needed.
- Never swallow exceptions silently; always log or propagate.

### Logging
Use a module-level logger: `logger = logging.getLogger(__name__)`.
Log progressions at `INFO`, skips/no-ops at `DEBUG`.

### Data Classes & Enums
- Use `@dataclass(frozen=True)` for immutable result objects.
- Inherit enums from `str`: `class ProgressionAction(str, Enum)` so values
  serialize directly as strings.

### Pydantic
Use Pydantic v2. Response models are defined in `routes.py` (no separate
schemas module). Use `model_config = ConfigDict(from_attributes=True)` on ORM
models.

### Linter Config (`pyproject.toml`)
```toml
[tool.ruff]
line-length = 100
target-version = "py311"

[tool.ruff.lint]
select = ["E", "F", "I", "N", "W", "UP"]
```

## TypeScript / Frontend Code Style

### Client Components
Every page file starts with `"use client";`. Server components are not used in
page files.

### API Layer
All TypeScript interfaces and the API client live in `web/lib/api.ts`. Add new
endpoint types and methods there. Use the `fetchApi<T>` generic helper for all
HTTP calls — it throws on non-2xx responses.

```ts
// Example
const data = await fetchApi<DashboardData>('/dashboard');
```

All API methods are collected on a single named `api` export object:
`api.getDashboard()`, `api.generateToday()`, etc.

### State Management
Plain `useState` + `useCallback`. No external state library (no Redux, Zustand,
etc.).

### Sub-components
Helper components used only within a page (e.g. `StatCard`) are defined in the
same file as the page, above the default export.

### Error Handling
```ts
try {
  // ...
} catch (e: unknown) {
  setError(e instanceof Error ? e.message : 'Unknown error');
}
```
Errors are displayed inline via state; never re-thrown to the top level.

### Naming
- Components: `PascalCase` function declarations
- Interfaces: `PascalCase`, semantically named (`PlanData`, `TodayExercise`)
- Variables / functions: `camelCase`
- API methods: `camelCase` verb phrases (`getDashboard`, `logToday`)

### Path Alias
`@/` maps to the `web/` root. Always use `@/lib/api` not relative paths.

### TypeScript Config
Strict mode is enabled (`"strict": true` in `tsconfig.json`). Do not disable
strict checks or use `any` without a comment explaining why.

## Design System

- **Dark mode only** — `<html>` always carries the `dark` class; never add
  light-mode variants.
- **Color palette:**
  - Background: `zinc-950` / `zinc-900` / `zinc-800`
  - Primary accent: `violet-500` / `violet-600`
  - Secondary accent: `indigo-400` / `indigo-500`
  - Success: `emerald-500` / `emerald-600`
  - Warning: `amber-400` / `amber-500`
  - Danger: `red-400` / `red-500`
- **Primary gradient:** `from-violet-600 to-indigo-500`
- **Card pattern:** `bg-zinc-800/60 border border-zinc-700/40 rounded-lg p-4 card-glow`
- **Fonts:** Inter (body), JetBrains Mono (numbers / monospace)
- **Responsive:** Mobile-first with `sm:` breakpoint. Mobile has a bottom tab
  bar (5 items); desktop uses a top nav bar (8 items).
- Use Tailwind utility classes exclusively — no CSS modules or styled-components.

## Testing Notes

- Framework: `pytest` + `pytest-asyncio` (`asyncio_mode = "auto"`)
- Test files live in `tests/`; name them `test_<module>.py`
- Use `@dataclass` mock objects instead of `MagicMock` when testing pure logic
- Tests should be class-based (`class TestFeatureName:`) with descriptive method
  names that read as sentences

## Key Design Constraints

- **Single-tenant:** `AthleteState` is always `id=1`. Do not add multi-user
  abstractions.
- **Deterministic progression:** The progression engine must be pure and
  deterministic — same inputs always produce the same output. Do not introduce
  randomness.
- **Anchors vs Staples:** Anchors are core exercises tracked for progression;
  Staples are accessory work. This distinction drives the progression engine.
- **LLM as planner only:** GPT-4o generates plans; it never directly mutates the
  database. All writes go through services.
